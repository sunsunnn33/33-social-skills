# 33-social-learn

分析 `publish-log.md` 中的发布记录，提炼账号规律，生成更新提案，经用户确认后写入 `ACCOUNT.md`。

---

## Language

用中文回复。文件路径和字段名保持英文。

---

## 文件角色说明

| 文件 | 由谁写 | 由谁读 | 作用 |
|------|--------|--------|------|
| `publish-log.md` | `33-social-publish` | 本 skill | 每次发布结果的原始记录 |
| `ACCOUNT.md` | 本 skill | `33-social-content` | 账号画像 + 积累的平台经验 |

---

## Step 0 — 定位文件

询问用户本次要复盘的项目文件夹（如不提供则询问）。

检查以下文件是否存在：

```
{folder}/publish-log.md          ← 本次发布记录（优先）
~/.33-skills/publish-log.md      ← 全局日志（如存在）

{folder}/ACCOUNT.md              ← 项目级账号记忆（优先）
~/.33-skills/ACCOUNT.md          ← 全局账号记忆（fallback）
```

**情况 A — publish-log.md 存在，ACCOUNT.md 存在：** 正常进入 Step 1。

**情况 B — publish-log.md 存在，ACCOUNT.md 不存在：**
```
还没有账号记忆文件，先建立一个基础档案。

（进入 Step 0b — 账号初始化）
```

**情况 C — publish-log.md 不存在：**
```
没有找到 publish-log.md，无法分析发布记录。

可以：
1. 提供包含 publish-log.md 的文件夹路径
2. 或直接告诉我这次发布的情况，我来手动记录并分析
```
如用户选择手动描述，进入 Step 1b（人工输入模式）。

---

## Step 0b — 账号初始化（首次运行）

如果 ACCOUNT.md 不存在，先问几个问题，建立基础档案：

```
没有找到账号档案，先建立一个基础档案。回答几个问题（可以随时补充或修改）：

1. 你的内容定位是什么？（例：科技开箱 / 生活记录 / 知识分享 / 美食探店）
2. 主要目标受众？（例：25-35岁职场人 / 科技爱好者 / 宝妈群体）
3. 你的内容风格？（例：干货硬核 / 轻松幽默 / 真实生活流）
4. 有没有绝对不碰的话题或禁用词？（例：不谈政治 / 不做对比测评 / 不说"最好"）
5. 目前哪些平台是重点？
```

收到回答后，创建初始 ACCOUNT.md（使用 Step 7 中的模板），然后继续 Step 1。

---

## Step 1 — 读取并分析 publish-log.md

读取 `publish-log.md` 全部内容。

提取每条记录的关键字段：
- 发布时间、平台、状态（已发布 / 跳过 / 失败）
- 标题、话题/标签
- 定时时间（如有）
- 备注（失败原因、特殊情况）

统计：
```
发布记录概览：
总计：{N} 条记录，覆盖 {M} 个平台
各平台：小红书 {n} 次 / 抖音 {n} 次 / ...
成功率：{x}%
时间范围：{first} → {last}
```

---

## Step 1b — 人工输入模式

如果没有 publish-log.md，引导用户描述：

```
告诉我这次发布的情况，我来帮你整理：

- 发布了哪些平台？
- 每个平台用的标题 / 描述是什么？
- 发布结果如何？（有没有异常、哪些顺利）
- 有没有临时改动？（改过什么，为什么改）
```

将用户描述结构化为标准 log 格式，然后继续 Step 2。

---

## Step 2 — 结合用户反馈

询问用户对这次发布是否有额外观察：

```
基于以上记录，有什么补充吗？

例如：
- 哪个平台发完感觉文案差了点
- 某个标签搜索不到被跳过了
- 发布顺序或时间上有什么想法
- 或者直接说"没有，继续"
```

如用户说没有补充，直接进入 Step 3。

---

## Step 3 — 提炼规律（Reflexion 格式）

对照 publish-log.md 内容 + 用户反馈，识别以下类型的信号：

**A. 操作问题**（流程上出了什么差错）
- 某平台上传卡住 / 封面失败 / 话题搜不到

**B. 内容判断**（文案层面的观察）
- 标题选了 A 还是 B，为什么
- 哪个平台的描述临时被修改过
- 标签被替换或跳过

**C. 时间和节奏**
- 是否按最佳时间发布，是否有定时
- 哪些平台发了，哪些跳过了，原因

**D. 平台特殊情况**
- 审核中、限流迹象、新功能发现

对每条有意义的信号，用 Reflexion 格式整理：

```
[信号类型] 平台：{platform}
发生了什么：{描述}
可能原因：{推断}
下次怎么做：{具体建议}
触发条件：{什么情况下这条建议生效}
```

---

## Step 4 — 读取当前 ACCOUNT.md

读取现有 ACCOUNT.md 内容，了解当前已有的账号记忆。

判断哪些 Step 3 的信号是：
- **新增信息**（ACCOUNT.md 里没有的）
- **强化信息**（印证了已有记录，可以提升置信度）
- **修正信息**（与已有记录矛盾，需要更新）
- **噪声**（单次偶发，不值得写入）

只有**新增 + 强化 + 修正**类信号才生成更新提案。

---

## Step 5 — 生成更新提案

展示建议写入 ACCOUNT.md 的内容，**逐条列出，不自动写入**：

```
=== ACCOUNT.md 更新提案 ===

【提案 1】平台经验 › 小红书 › 标签
操作：新增
内容：话题「#咖啡」在小红书搜索不到，改用「#咖啡日记」成功添加
建议：{topic} 搜不到时，尝试加「日记」「分享」「体验」后缀

采纳 / 跳过 / 修改？

---

【提案 2】平台经验 › 抖音 › 最佳发布时间
操作：更新（原：18:30-20:00）
内容：本次 19:00 发布，与建议时间一致，无异常
建议：维持现有建议，该时段有效

采纳 / 跳过 / 修改？

---

（共 {N} 条提案）
```

---

## Step 6 — 用户逐条确认

等待用户对每条提案的反馈：
- **采纳**：写入 ACCOUNT.md
- **跳过**：不写入，不保留
- **修改**：用户提供修改后内容，确认后写入
- **全部采纳**：一次性写入所有提案

所有提案处理完毕后，进入 Step 7。

## 关于学习节奏

**建议的调用时机：**
- 每次 `33-social-publish` 完成后立即调用（趁记忆新鲜）
- 每发布 3-5 次后做一次「批量复盘」（找跨多次的规律）

**不要每次都调用的情况：**
- 发布完全正常，没有任何异常或修改 → 可跳过，数据不够多时学不到新东西

**ACCOUNT.md 不是越多越好：**
- 单次偶发的情况不写入
- 同一条规律写入一次就够，不重复堆砌
- 矛盾的新数据出现时，用新数据覆盖旧记录，不保留两个版本
